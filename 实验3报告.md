## 二、流程理解

`Pack::decode` 是 pack 文件解码的主入口。它先通过 `Pack::check_header` 校验 `PACK` 魔数、版本号和对象数量，然后循环读取每个 pack object。每个对象经过类型和大小解析、zlib 解压后被封装为 `CacheObject`。

普通对象会直接写入 cache，并触发等待它的 delta 对象继续处理。delta 对象会根据 base object 的 offset 或 hash 查找 cache；如果 base object 尚未完成，则临时放入 `Waitlist`。当 base object 完成后，`process_waitlist` 再递归处理等待队列中的 delta。`Caches` 负责对象缓存和必要时的落盘，`Waitlist` 负责保存尚未满足依赖的 delta 对象。

主 Decode 循环中还有一个背压判断：当线程池中排队任务过多，或者内存使用超过限制时，解码线程会暂停并 `yield_now`，避免继续读取 pack 导致队列和缓存压力持续扩大。

## 三、原代码问题

本次选择的改进点不是改变解码语义，而是提升背压策略代码的可读性和可测试性。原实现中存在两个维护性问题：

1. Decode 主循环中直接硬编码线程池队列阈值 `2000`，阈值含义不够明确。
2. “是否需要暂停解码”的判断内联在 `while` 条件中，依赖队列长度、内存使用量和 `mem_limit`，但没有独立函数，难以单独测试边界行为。
3. `Pack::new` 中“总内存限制的 80% 分配给 cache”的计算直接写在构造函数中，缺少命名语义。

## 四、修改内容

### 1. 提取 Decode 背压常量

在 `src/internal/pack/decode.rs` 中新增：

```rust
const DECODE_QUEUE_BACKPRESSURE_LIMIT: usize = 2000;
const CACHE_MEMORY_LIMIT_NUMERATOR: u128 = 4;
const CACHE_MEMORY_LIMIT_DENOMINATOR: u128 = 5;
```

`DECODE_QUEUE_BACKPRESSURE_LIMIT` 表示 Decode 线程池允许积压的任务上限；另外两个常量用于表达 cache 使用总内存限制的 `4/5`。

### 2. 提取缓存内存计算函数

新增纯函数：

```rust
fn cache_memory_limit(mem_limit: usize) -> usize {
    ((mem_limit as u128) * CACHE_MEMORY_LIMIT_NUMERATOR / CACHE_MEMORY_LIMIT_DENOMINATOR) as usize
}
```

`Pack::new` 改为：

```rust
let cache_mem_size = mem_limit.map(cache_memory_limit);
```

这样保留原有 80% 分配策略，同时继续使用 `u128` 中间计算，避免 32 位平台上的乘法溢出风险。

### 3. 提取暂停解码判断函数

新增纯函数：

```rust
fn should_pause_decode(queued_tasks: usize, memory_used: usize, mem_limit: Option<usize>) -> bool {
    queued_tasks > DECODE_QUEUE_BACKPRESSURE_LIMIT
        || mem_limit.map(|limit| memory_used > limit).unwrap_or(false)
}
```

主循环改为调用该函数：

```rust
while should_pause_decode(self.pool.queued_count(), self.memory_used(), self.mem_limit) {
    thread::yield_now();
}
```

这使 Decode 主流程更集中，背压策略也可以不依赖真实线程池和真实 cache 进行单元测试。

### 4. 兼容本地 Rust 工具链的等价改写

本机环境为 `rustc 1.86.0`。源码中原有两处 `if let ... && let ...` 语法在该工具链下无法编译，因此将其改为等价的稳定 Rust 写法：

- `src/internal/pack/cache_object.rs`：将 `Drop` 中的 let-chain 改为提前返回和 `let Some(...) = ... else`
- `src/internal/pack/decode.rs`：将 `decode_stream` 中发送 pack id 的 let-chain 改为嵌套 `if let`

该修改不改变运行逻辑，只用于保证当前环境可以构建和测试。

## 五、新增测试

在 `src/internal/pack/decode.rs` 的测试模块中新增 4 个单元测试：

1. `test_cache_memory_limit_uses_configured_ratio`
   - 验证 cache 内存上限按 `4/5` 计算。
   - 覆盖 `usize::MAX`，确认使用 `u128` 中间计算不溢出。

2. `test_should_pause_decode_when_queue_exceeds_limit`
   - 当排队任务数等于阈值时不暂停。
   - 当排队任务数超过阈值时暂停。

3. `test_should_pause_decode_when_memory_exceeds_limit`
   - 当内存使用等于 `mem_limit` 时不暂停。
   - 当内存使用超过 `mem_limit` 时暂停。

4. `test_should_not_pause_decode_without_memory_limit`
   - 当 `mem_limit = None` 时，即使传入很大的 memory_used，也不会因为内存条件暂停。

这些测试覆盖了实验要求中的队列超过阈值、内存超过限制、`mem_limit = None` 等边界情况。

## 七、总结

本次实验围绕 Pack Decode 主循环中的背压策略做了小范围重构：将硬编码阈值和内存比例提取为命名常量，将暂停判断提取为纯函数，并通过 4 个单元测试覆盖关键边界。修改后主 Decode 流程的语义保持不变，但策略代码更清晰，也更容易在不构造完整 pack 文件的情况下验证。
